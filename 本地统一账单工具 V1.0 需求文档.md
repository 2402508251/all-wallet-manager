# 本地统一账单工具 V1.0 功能需求文档

## 一、项目定位

本地统一账单工具是一款纯本地单机记账与账务整理工具。系统不依赖云端服务，不上传用户隐私数据，主要用于统一归集微信、支付宝、建设银行等渠道账单，解决多渠道流水分散、账户归属不清、跨平台支付重复统计、信用消费与还款混淆、转账影响报表口径等问题。

V1.0 的目标是形成完整的本地账单处理闭环：采集账单、解析标准化、自动归属、分类统计、账务处理、报表查看，以及基于源账单和快照的数据维护能力。

## 二、产品范围

### 2.1 范围内能力

1. 本地文件上传与邮箱 IMAP 主动拉取账单。
2. 微信、支付宝、建设银行账单解析与统一标准化。
3. 账户、角色、家庭维度的账单归属管理。
4. 账单查询、详情查看、编辑、批量处理、逻辑删除与回收站恢复。
5. 分类规则配置、自动分类与重新分类。
6. 跨平台支付合并、转账配对、信用消费与还款识别。
7. 月度收支、分类分布、趋势和排行统计。
8. 源账单保存、重新解析、快照恢复与数据清理。

### 2.2 范围外能力

1. 不提供云端同步、多人在线协作或远程备份。
2. 不直接连接微信、支付宝、银行官方接口，只处理用户导出或邮箱收到的账单文件。
3. 不做投资组合管理、预算审批、发票报销等企业财务功能。
4. 不支持一笔账单按比例拆分到多个家庭或角色。

## 三、功能需求总览

### FR-01 账单采集

#### FR-01-01 本地文件上传

1. 支持用户手动选择或拖拽上传账单文件。
2. 支持常见账单格式：`.xlsx`、`.csv`、`.xls`、`.pdf`、`.zip`。
3. 上传后的文件进入采集记录列表，用户可查看文件名、来源、渠道、状态和解析结果。
4. 系统根据文件名等信息自动识别渠道；识别失败时标记为未知渠道，允许用户手动修正。

#### FR-01-02 邮箱 IMAP 拉取

1. 支持用户配置邮箱 IMAP 信息，并由用户主动点击同步账单。
2. 邮箱拉取不做后台定时轮询，不在用户未触发时读取邮件。
3. 系统按发件人白名单、主题关键词和附件类型过滤账单邮件。
4. 支持基于 UID 或拉取时间的增量拉取，避免重复处理历史邮件。
5. 支持保存多个邮箱配置、测试连接、清除授权凭证和删除邮箱配置。
6. 邮箱授权码等敏感信息需要加密存储。

#### FR-01-03 ZIP 与加密附件处理

1. 支持从 ZIP 压缩包中提取账单文件。
2. 遇到加密 ZIP 时提示用户输入解压密码。
3. 解压后仅处理支持的账单文件，忽略无关附件。
4. 采集和解析过程需要向用户展示进度，避免耗时操作被误认为卡死。

#### FR-01-04 采集记录管理

1. 每次上传或邮箱拉取生成采集记录。
2. 采集记录需保存来源、文件路径、文件名、渠道、状态、解析结果、关联批次等信息。
3. 用户可对采集记录执行解析、删除采集记录、删除关联账单等操作。
4. 解析结果需展示成功、重复、待分配、未分类等统计信息。

### FR-02 账单解析与标准化

#### FR-02-01 多渠道解析

1. 支持微信账单解析，覆盖 `.xlsx`、`.csv` 等导出格式。
2. 支持支付宝账单解析，覆盖 `.csv`、`.xlsx` 等导出格式。
3. 支持建设银行账单解析，覆盖 `.xls`、`.pdf` 等格式。
4. 解析器应能处理表头位置变化、编码差异、退款/关闭交易等常见账单差异。
5. 建设银行账单无稳定全局交易号时，系统需要生成可去重的内部交易标识。

#### FR-02-02 统一账单模型

1. 各渠道原始字段需映射到统一账单模型。
2. 交易时间统一为标准时间格式。
3. 金额统一以整数分存储。
4. 收支方向、交易类型、支付方式等渠道枚举需映射为系统内部枚举。
5. 系统需保存源账单原始字段，便于追溯和重新解析。

#### FR-02-03 去重、批次与事务

1. 系统基于渠道和渠道交易号进行去重，重复账单跳过导入。
2. 每次导入生成唯一批次，用于追踪导入结果和按批次回滚。
3. 批量导入写库需要事务保护，异常时避免产生半完成数据。

### FR-03 账单查询与维护

#### FR-03-01 账单列表与筛选

1. 用户可按时间、渠道、账户、角色、分类、关键词等条件查询账单。
2. 账单列表需支持分页展示。
3. 报表页面可下钻到账单列表，查看对应分类或统计口径下的明细。

#### FR-03-02 账单详情与编辑

1. 用户可查看账单基础信息、账务信息和源账单信息。
2. 用户可编辑允许手动维护的账单字段，例如分类、备注、归属相关信息等。
3. 手动编辑过的账单需要标记，避免重新解析时被无意覆盖。
4. 支持单条或批量更新账单信息。

#### FR-03-03 删除与回收站

1. 普通删除采用逻辑删除，账单进入回收站。
2. 回收站是账单管理页的一部分，不作为独立导航页面。
3. 用户可在回收站中恢复账单、永久删除账单或清空回收站。
4. 永久删除需同步清理与该账单强关联的源账单、账务状态等数据。

### FR-04 归属分配与账户管理

#### FR-04-01 账户自动提取

1. 导入账单时，系统应从账单文件和交易记录中提取账户标识。
2. 微信账单需识别昵称和支付方式，并归并为微信账户、银行卡、信用账户、余额或优惠类账户。
3. 支付宝账单需识别账户尾号和支付方式，并归并为支付宝账户、银行卡、信用账户、余额或优惠类账户。
4. 建设银行账单需识别卡号尾号等账户标识。
5. 无法确认归属时，账单应进入默认未分配角色或待分配状态。

#### FR-04-02 家庭、角色与账户关系

1. 系统支持家庭、角色和账户三类基础资料管理。
2. 一个角色可以关联一个或多个家庭。
3. 一个账户归属于一个角色。
4. 账单通过账户间接确定角色归属。
5. 用户可在系统设置中调整账户所属角色，以及角色与家庭的关联关系。

#### FR-04-03 账户别名与合并

1. 系统支持为账户维护别名，解决同一账户在不同账单中名称不一致的问题。
2. 微信账户支持基于昵称别名进行识别和归并。
3. 用户可合并重复账户，也可在必要时取消合并。

### FR-05 账务处理

#### FR-05-01 跨平台支付合并

1. 系统需要识别微信、支付宝等第三方平台使用银行卡或他人代付时产生的跨平台重复流水。
2. 匹配成功后不删除任何原始账单，只建立合并关系。
3. 统计时需区分发起方与真实支付者：角色层面可按消费发起方统计，账户层面可按实际扣款账户统计。
4. 单边记录暂未匹配到对端时，可标记为待合并或孤立记录。
5. 用户可手动确认孤立记录为独立消费，或撤销已建立的合并关系。

#### FR-05-02 转账配对

1. 系统应识别账户间转出与转入流水，并建立转账关联。
2. 强匹配记录可自动配对；弱匹配候选需要用户确认。
3. 转账手续费可作为系统记录单独展示或关联。
4. 报表统计中应避免内部转账影响真实消费/收入口径。

#### FR-05-03 信用消费与还款追踪

1. 系统需识别花呗、分付、信用卡等信用消费。
2. 信用消费需要关联到对应信用账户。
3. 还款流水需要与信用账户或原消费关系建立关联。
4. 系统可生成还款镜像等系统记录，用于账务展示和统计口径处理。
5. 报表支持隐藏还款转账等内部流转影响。

#### FR-05-04 建设银行账单特殊处理

1. 建设银行账单解析时需要处理无全局唯一交易号的问题。
2. 系统应从卡号尾号、交易时间、金额、描述等信息中提取可用于匹配和归属的字段。
3. 建行流水可作为跨平台真实支付方参与合并匹配。

### FR-06 统计报表

#### FR-06-01 月度汇总

1. 系统提供月度收入、支出、结余等摘要指标。
2. 统计口径需排除逻辑删除账单。
3. 系统记录、还款转账、内部转账等是否计入统计，应按账务处理结果和用户设置控制。

#### FR-06-02 分类与趋势分析

1. 系统提供按分类的消费分布统计。
2. 系统提供按时间的收入、支出趋势统计。
3. 系统提供分类排行，便于定位主要消费方向。
4. 报表图表需支持用户查看对应明细账单。

#### FR-06-03 家庭与角色口径

1. 报表需支持按家庭或角色维度筛选。
2. 对跨平台合并记录，系统需在账户口径和角色口径上保持一致、可解释的统计规则。

### FR-07 设置与数据管理

#### FR-07-01 基础资料设置

1. 家庭管理：创建、编辑、删除家庭。
2. 角色管理：创建、编辑、删除角色，并维护角色与家庭关系。
3. 账户管理：创建、编辑、删除账户，维护账户归属、别名和合并关系。
4. 分类管理：维护账单业务分类。
5. 邮箱配置：维护邮箱 IMAP 配置与授权凭证。

#### FR-07-02 分类规则与重新分类

1. 用户可维护分类关键词和匹配字段。
2. 分类规则支持不同字段、优先级和匹配策略。
3. 用户修改分类规则后，可对历史账单执行重新分类。
4. 手动修改过分类的账单应避免被自动重新分类覆盖，除非用户明确选择覆盖。

#### FR-07-03 源账单与重新解析

1. 导入时需保存源账单原始 JSON 数据。
2. 用户修改解析映射、枚举配置或系统规则后，可基于源账单重新解析。
3. 重新解析应支持选择范围，例如全部、指定批次、指定渠道或指定时间范围。
4. 重新解析前应创建快照，便于回退。
5. 用户可清理源账单数据以释放存储空间；清理后相关账单将无法再基于源数据重新解析。

#### FR-07-04 快照与数据清理

1. 系统在重新解析、重新分类、批量归属调整等高影响操作前创建快照。
2. 用户可查看快照列表和快照详情。
3. 用户可从快照恢复数据。
4. 用户可清理过期快照。
5. 用户可按采集记录清理关联账单，也可清空全部账单数据。

## 四、非功能需求

### NFR-01 本地与隐私

1. 所有账单数据、配置和快照默认存储在本地。
2. 系统不上传账单、邮箱授权码或其他隐私数据到云端。
3. 邮箱授权码等敏感信息需要加密保存。

### NFR-02 可追溯与可回退

1. 关键导入数据需保留源账单，便于用户查看原始字段。
2. 高影响批量操作前需创建快照。
3. 删除默认采用逻辑删除，给用户恢复机会。

### NFR-03 数据一致性

1. 批量导入、重新解析、重新分类、批量归属调整等写操作需要事务保护。
2. 使用事务时需避免嵌套事务导致数据库锁或部分提交问题。
3. 数据库结构以初始化脚本为准，结构变更时不做复杂历史迁移。

### NFR-04 交互与性能

1. 上传、解析、邮箱拉取等耗时操作需要展示进度。
2. 账单列表需要分页，避免一次性加载大量数据。
3. 桌面端界面应保持轻量，不引入不必要的复杂技术栈。

### NFR-05 可维护性

1. 前端通过桥接封装调用后端能力。
2. 后端按采集、解析、标准化、归属、分类、账务、重解析等模块组织。
3. 需求文档描述用户可感知能力，设计文档描述系统结构和关键数据关系。

## 五、术语表

| 术语/缩写 | 解释 |
| :--- | :--- |
| 统一账单 | 各渠道原始账单经解析和标准化后形成的系统内部账单记录。 |
| 源账单 | 导入时保存的原始账单字段 JSON，用于溯源和重新解析。 |
| 账户 | 系统中的资金或支付载体，例如微信账户、支付宝账户、银行卡、信用账户等。 |
| 角色 | 消费或资金归属主体，例如本人、配偶、公用角色等。 |
| 家庭 | 报表统计和归属管理的家庭维度，一个角色可关联多个家庭。 |
| 业务分类 | 统计层分类，例如餐饮美食、交通出行、购物消费等。 |
| trade_type | 账务层交易类型，例如消费、退款、转账、还款、手续费等。 |
| transfer_link_id | 转账关联号，用于关联一笔转出与一笔转入。 |
| merged_group_id | 跨平台合并组 ID，用于关联发起方记录和真实支付方记录。 |
| 真实支付者 | 跨平台支付中实际扣款的账户。 |
| 发起方 | 跨平台支付中发起交易的账户或角色。 |
| 孤立记录 | 跨平台合并场景中暂未找到对端真实支付记录的单边账。 |
| 逻辑删除 | 将账单标记为删除并隐藏在正常列表中，但仍允许从回收站恢复。 |
| 快照 | 高影响数据操作前保存的备份记录，用于查看差异和恢复。 |

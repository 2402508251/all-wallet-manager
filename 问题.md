# 具体设计文档审计报告

> 审计对象：《本地统一账单工具 V0.5 具体设计文档》
> 审计维度：设计合理性 + 逻辑一致性

---

## 一、架构设计层面

### ✅ 合理之处

- PyWebView + Vue3 + SQLite 的桌面应用技术栈选型合理，单文件数据库便于本地工具分发
- JS↔Python 双向通信机制设计清晰，异步任务通过 task_id + 事件推送的模式避免了阻塞
- 前后端分离（Vite 构建 → dist → PyWebView 加载）是成熟方案

### ⚠️ 问题

**问题 1：PyWebView 事件推送 API 不准确**

文档中 Python→JS 使用 `window.emit('event_name', data)`，但 PyWebView 实际的 Python→JS 推送 API 是 `webview.windows[0].evaluate_js()` 或 `window.pywebview.api` 回调。`window.emit()` 并非 PyWebView 原生机制，需自行实现或使用第三方库（如 `pywebview.event`）。建议明确底层实现方式。

- 严重程度：🟢 低
- 影响范围：实现可行性

---

**问题 2：SQLite 多线程写入安全未充分设计**

文档提到耗时操作使用 `threading.Thread`，但 SQLite 在 WAL 模式下虽支持并发读，写入仍是串行的。多个线程同时执行导入/解析/快照操作时，需明确：

- 是否启用 WAL 模式？
- 连接是否线程局部（`check_same_thread=False`）？
- 写入锁超时策略（`busy_timeout`）？

`DatabaseManager.get_connection()` 标注为"获取线程局部连接"，但文档未说明具体实现。

- 严重程度：🟢 低
- 影响范围：运行稳定性

---

**问题 3：进程通信缺少超时与错误恢复**

异步任务返回 `{ task_id }` 后，若 Python 线程异常崩溃，前端将永远收不到 `task_done` 事件。缺少：

- 任务超时检测机制
- 任务状态持久化（进程重启后任务丢失）
- 前端侧的任务超时兜底逻辑

- 严重程度：🟢 低
- 影响范围：用户体验

---

## 二、数据库设计层面

### ✅ 合理之处

- unified_bills + source_bills 分离设计，核心表精简、原始数据可溯源
- bill_accounting 拆分为扩展表，避免核心表字段膨胀
- (channel, channel_trade_no) 唯一索引用于去重，设计正确
- 快照采用增量 Diff 两表存储，节省空间

### ⚠️ 问题

**问题 4：`roles` 表 `role_type` 与 `is_shared` 字段冗余**

`role_type`（personal/shared）和 `is_shared`（0/1）表达的是同一语义。当 `role_type='shared'` 时 `is_shared` 必然为 1，两者存在一致性风险。建议移除 `is_shared`，仅保留 `role_type`。

- 严重程度：🟡 中
- 影响范围：维护性

---

**问题 5：`unified_bills.family_id` 和 `role_id` 反范式化缺少级联更新策略** 🔴

文档明确 `role_id` 从 `account_id → accounts.role_id` 派生，`family_id` 从角色的主要家庭派生。但存在以下问题：

- **修改账户角色时**：`update_account` 修改 `role_id` 后，文档仅提到调用 `reassign_pending_bills` 处理 `assign_status='pending'` 的账单，但**已分配账单**（`assign_status='assigned'`）的 `role_id` 和 `family_id` 也需要级联更新，文档未覆盖此场景。
- **修改角色主要家庭时**：所有关联账单的 `family_id` 都需要更新，文档完全没有提及此级联操作。
- **删除家庭/角色时**：外键约束会阻止删除，但文档未定义 `ON DELETE` 策略（CASCADE/RESTRICT/SET NULL）。

- 严重程度：🔴 高
- 影响范围：数据一致性

---

**问题 6：`collection_records.account_id` 一对多矛盾**

一个采集记录（如微信账单文件）可能包含多种支付方式（零钱、银行卡等），对应多个账户。但 `collection_records.account_id` 是单值外键，只能关联一个账户。建议改为多对多关联或移除此字段（账单级别的 `account_id` 已足够）。

- 严重程度：🟡 中
- 影响范围：数据模型

---

**问题 7：`bill_accounting.merged_to_id` 撤销合并时原始数据丢失** 🔴

`cross_platform_merger.py:undo_merge` 中撤销合并需"恢复银行卡记录的原始 product_desc / counterparty"，但合并时用第三方记录覆盖了银行卡记录的这些字段，**原始值未在任何地方保存**。快照引擎可以解决此问题，但 `undo_merge` 的流程描述中没有调用快照恢复，而是直接恢复。建议在合并时将原始值存入 `bill_accounting` 表或依赖快照。

- 严重程度：🔴 高
- 影响范围：数据正确性

---

**问题 8：`delete_bill` 硬删除与外键约束冲突** 🔴

API 文档中 `delete_bill` 标注为"硬删除"，但 `unified_bills` 被以下表外键引用：

- `source_bills.bill_id`
- `bill_accounting.bill_id`
- `snapshot_details.bill_id`

硬删除会导致外键冲突或孤儿记录。文档未定义级联删除策略。

- 严重程度：🔴 高
- 影响范围：数据完整性

---

**问题 9：快照恢复与已删除记录的冲突**

`snapshot_details.bill_id` 引用 `unified_bills.id`。如果账单已被硬删除，快照恢复时 `UPDATE unified_bills SET ... WHERE id = ?` 将找不到目标行，恢复静默失败。文档未处理此场景。

- 严重程度：🟢 低
- 影响范围：边界场景

---

**问题 10：`is_default` 家庭缺少唯一约束**

`families` 表的 `is_default` 字段允许多行为 1，即可以有多个"默认家庭"。文档说"取消原默认"，但没有数据库层面的约束保障，并发操作可能导致多个默认家庭。

- 严重程度：🟡 中
- 影响范围：数据一致性

## 三、模块设计层面

### ✅ 合理之处

- 解析器工厂模式 + 基类抽象，扩展新渠道方便
- 标准化引擎拆分为字段映射/时间/金额/枚举四个子模块，职责清晰
- 跨平台合并的三种场景（先第三方/先银行卡/二次扫描）覆盖完整

### ⚠️ 问题

**问题 11：跨平台合并逻辑不对称 — 银行卡记录无法被后续高效匹配**

`cross_platform_merger.py:try_merge` 中，银行卡记录导入时若无匹配的孤儿记录，则"银行卡记录无需创建 bill_accounting 记录"。这意味着：

- 银行卡记录没有 `bill_accounting` 条目
- 后续导入第三方支付时，`mark_orphan()` 搜索的是"金额一致+时间接近的银行卡记录"，但搜索范围是全表扫描 `unified_bills`，而非通过 `bill_accounting.merge_status` 索引
- 性能问题：随着数据量增长，全表扫描效率低下

建议：银行卡记录也创建 `bill_accounting` 记录（`merge_status='normal'`），便于索引查询。

- 严重程度：🟡 中
- 影响范围：查询性能

---

**问题 12：`CreditTracker.link_repayment` 还款匹配逻辑模糊**

文档说"查找对应 credit_account（还款来源账户 + 信用账户）"，但：

- 一笔还款的 `payment_method` 可能只写"建设银行储蓄卡"，如何确定还的是哪个信用账户？
- 一个人可能有多个信用账户（花呗+建行信用卡），仅靠"还款来源账户"无法唯一确定目标信用账户
- 缺少匹配优先级和冲突处理策略

- 严重程度：🟡 中
- 影响范围：功能正确性

---

**问题 13：弱匹配候选无去重与拒绝持久化**

`transfer_pairer.auto_pair_weak_candidates()` 返回候选列表后：

- 用户拒绝后，下次查询同一记录仍会返回该候选（无拒绝记录）
- 同一对候选可能被重复推荐
- 建议增加 `bill_accounting` 的拒绝状态或独立的拒绝记录表

- 严重程度：🟡 中
- 影响范围：用户体验

---

**问题 14：快照引擎的两阶段语义不清晰** 🔴

`SnapshotEngine.create_snapshot()` 的描述是"逐表逐条比对变更字段"，但快照是在操作**之前**创建的，此时只有"当前值"（即 old_value），new_value 应在操作**之后**填充。文档描述的 `create_snapshot` 似乎试图一次性完成，逻辑上不可能。实际流程应为：

1. 操作前：记录 old_value
2. 执行操作
3. 操作后：记录 new_value

但文档的 `create_snapshot` 方法签名只接收 `affected_records`，没有"操作后回填"的步骤。

建议重构为两阶段：
- `create_snapshot()` → 记录 old_value
- `finalize_snapshot()` → 记录 new_value
- 或改为操作完成后创建"反向快照"（记录当前值作为恢复目标）

- 严重程度：🔴 高
- 影响范围：快照功能可用性

---

**问题 15：~~重新解析与账务处理的交互未定义~~** ✅ **已修复**

修复内容：
- `ReParser.reparse()` 方法新增步骤 7（收集 ReparseChange 列表）和步骤 8（调用 `_cascade_accounting_updates` 处理级联更新）
- 新增 `ReparseChange` 数据类，记录每条账单变更前后的字段值

- 前端交互流程增加级联影响提示（合并关系解除/配对关系解除）
- data store 的 reparse action 增加级联统计解析和 ElNotification 提示逻辑

- ~~严重程度：🔴 高~~ ✅ 已修复

**问题 16：建行 channel_trade_no 生成策略有碰撞风险**

`CCBHandler.generate_trade_no()` 格式为 `CCB_{account_tag}_{YYYYMMDD}_{4位序号}`，4位序号上限为 9999。如果同一天同账户超过 9999 笔交易会溢出。且序号的生成依赖"当天已有最大序号+1"，并发导入时可能产生重复。

- 严重程度：🟡 中
- 影响范围：去重可靠性

## 四、前端设计层面

### ✅ 合理之处

- Hash 路由模式适配 PyWebView 本地文件加载
- Pinia Store 拆分合理，依赖关系清晰
- ECharts 按需引入减小体积
- Bridge 通信封装统一错误处理

### ⚠️ 问题

**问题 17：Vue Router 4 导航守卫 API 使用错误**

文档中 `beforeLeave` 使用 `next(false)` 模式，这是 Vue Router 3 的 API。Vue Router 4（对应 Vue3）的导航守卫不再使用 `next()` 回调，而是通过返回值控制：

```javascript
// Vue Router 4 正确写法
beforeRouteLeave() {
  if (hasUnsavedChanges) {
    return confirm('有未保存的更改，确认离开？')
  }
}
```

- 严重程度：🟡 中
- 影响范围：代码正确性

---

**问题 18：Bridge 就绪检查在导航守卫中冗余**

`main.js` 中应用在 `ready()` resolve 后才 `mount`，因此导航守卫执行时 Bridge 必然已就绪。`beforeEach` 中再次检查 `bridge.isReady()` 是冗余逻辑。

- 严重程度：🟢 低
- 影响范围：代码质量

---

**问题 19：`useECharts` composable 缺少响应式更新**

`composables/useECharts.js` 中导入了 `watch` 但未使用。`updateChart()` 需要手动调用，没有自动响应数据变化的机制。建议增加 `watch` 对 `optionGetter` 的监听。

- 严重程度：🟢 低
- 影响范围：开发体验

---

**问题 20：report store 的 dateRange getter 时区问题** 🟡

```javascript
const end = new Date(year, month, 0).toISOString().slice(0, 10)
```

`toISOString()` 返回 UTC 时间，在 UTC+8 时区下，`new Date(2026, 6, 0)`（6月30日）的 UTC 表示可能是 `2026-06-29T16:00:00.000Z`，slice 后得到 `2026-06-29`，少了一天。应使用本地格式化：

```javascript
const end = new Date(year, month, 0)
const endStr = `${end.getFullYear()}-${String(end.getMonth()+1).padStart(2,'0')}-${String(end.getDate()).padStart(2,'0')}`
```

- 严重程度：🟡 中
- 影响范围：报表数据正确性

---

**问题 21：批量删除缺少原子性 API**

账单列表页的"批量删除"通过逐条调用 `delete_bill` 实现，无法保证原子性（部分成功部分失败）。建议增加 `batch_delete_bills` API，在事务中执行。

- 严重程度：🟢 低
- 影响范围：数据一致性

## 五、API 设计层面

### ⚠️ 问题

**问题 22：`upload_files` 参数设计不符合浏览器安全限制**

参数 `{ files: [{name, path, size}] }` 包含本地文件路径，但浏览器出于安全限制不允许 JS 获取文件完整路径。PyWebView 的文件选择应通过 Python 侧的 `webview.windows[0].create_file_dialog()` 实现，而非前端 `<input type="file">`。

- 严重程度：🟡 中
- 影响范围：功能可行性

---

**问题 23：缺少错误码体系**

API 返回 `{ success: false, message: "..." }` 只有文本描述，缺少结构化错误码（如 `DUPLICATE_BILL`, `ACCOUNT_NOT_FOUND`），前端无法做精细化错误处理和国际化。

- 严重程度：🟢 低
- 影响范围：可维护性

---

**问题 24：`toggle_hide_repayment_transfer` 状态持久化不明**

此 API 切换"隐藏还款转账"，但未说明状态存储在哪里。如果是内存态，重启后丢失；如果是数据库态，缺少对应的表/字段设计。

- 严重程度：🟢 低
- 影响范围：功能完整性

---

**问题 25：~~`clear_all_bills` 缺少快照保护~~ ✅ 已修复**

修复内容：`clear_all_bills` API 说明已增加"操作前自动创建快照"，与重新解析的处理方式一致。

## 六、配置管理层面

### ⚠️ 问题

**问题 26：分类关键词双存储 — 数据库与 JSON 文件不一致风险** 🟡

`category_keywords` 同时存在于：

- 数据库表 `category_keywords`（2.2.10）
- JSON 文件 `config/category_keywords.json`（8.1.1）

文档未明确哪个是权威数据源。`ConfigManager` 从 JSON 加载，但前端设置页通过 `save_category_keywords` API 保存到数据库。两者可能不同步。

建议：以数据库为唯一数据源，JSON 文件仅作为初始化种子数据。

- 严重程度：🟡 中
- 影响范围：数据一致性

---

**问题 27：`CredentialEncryptor` 机器特征值不可靠**

文档说"使用机器特征值派生密钥"，但：

- Windows 重装后机器 GUID 变化
- 网卡更换后 MAC 地址变化
- 上述情况会导致已加密的授权码无法解密

建议：增加恢复机制（如用户重新输入授权码），或在密钥派生中使用更稳定的特征（如主板序列号）并配合用户可导出的恢复密钥。

- 严重程度：🟢 低
- 影响范围：用户体验

---

**问题 28：ConfigManager 缓存无失效机制**

`ConfigManager` 使用内存缓存，但：

- 用户在设置页修改配置后，缓存何时更新？
- 如果 `data/wallet.db` 被外部工具修改，缓存与实际不一致
- 建议在 `save()` 时更新缓存，并考虑文件监听（`watchdog`）机制

- 严重程度：🟢 低
- 影响范围：数据一致性

## 八、优先修复建议

## 九、前端设计深度审计

### 9.1 路由与页面结构

**问题 29：路由缺少 404 兜底路由**

路由表中定义了 5 条路由 + 1 条重定向，但没有 `{ path: '/:pathMatch(.*)*', ... }` 兜底路由。用户手动输入错误路径时将看到空白页面而非友好提示。

- 严重程度：🟢 低
- 影响范围：用户体验

---

**问题 30：路由切换时 Store 状态未隔离**

5 个页面共享 Pinia Store，但 Store 的 state 是全局单例。例如：
- 用户在 `/bills` 页面设置了筛选条件，切换到 `/accounting` 再切回来，筛选条件仍保留（可能期望重置）
- 用户在 `/reports` 选择了月份，切换到其他页面再回来，月份状态保留（可能期望回到当月）

文档未定义路由切换时的 Store 状态重置策略。建议在 `beforeRouteLeave` 或 `onUnmounted` 中按需重置，或使用 `defineStore` 的 `reset` 功能。

- 严重程度：🟡 中
- 影响范围：用户体验

---

**问题 31：路由 query 参数类型不一致**

路径 3（图表交互→路由跳转）中 `router.push` 传递 `category_id` 为数字，但 `route.query` 取出时始终为字符串。`billStore.filter = { ...route.query }` 直接展开后，`category_id` 从数字变为字符串，后端 API 期望 `int` 类型时可能出错。

建议在 BillListView 的 `onMounted` 中对 `route.query` 做类型转换。

- 严重程度：🟡 中
- 影响范围：功能正确性

---

### 9.2 Pinia Store 设计

**问题 32：system store 职责过重 — 违反单一职责原则**

system store 承担了 7 个 Tab 的全部状态和操作（家庭/角色/账户/分类/邮箱/关键词/数据管理），actions 数量达 28 个。问题：

- 任何一个 Tab 的状态变更都可能触发无关组件的重新渲染
- 代码维护困难，单个 store 文件将非常庞大
- 与其他 Store 的依赖关系中，bill/collection/accounting/report 都依赖 system，形成星型耦合

建议拆分为：`familyStore`、`roleStore`、`accountStore`、`categoryStore`、`emailStore`、`dataStore`。

- 严重程度：🟡 中
- 影响范围：可维护性、性能

---

**问题 33：accounting store 的 `hide_repayment` 与 report store 的 `hide_repayment` 重复**

两个 store 各自维护了 `hide_repayment` 状态，但文档中只有一个 `toggle_hide_repayment_transfer` API。两个 store 的状态可能不一致。建议统一为一个全局状态（如 system store 或独立的 UI 状态 store）。

- 严重程度：🟡 中
- 影响范围：状态一致性

---

**问题 34：accounting store 的 `getCreditRecords` 和 `getRepayRecords` 复用 query_bills 但缺少分页**

文档注释说"复用 query_bills + 筛选"，但 accounting store 的 state 中 `credit_records` 和 `repay_records` 是数组，没有 `page/page_size/total`。如果信用消费记录很多（如花呗每月数百笔），一次加载全部数据会导致性能问题。

- 严重程度：🟡 中
- 影响范围：性能

---

**问题 35：bill store 的 `deleteBills` 逐条调用缺少进度反馈**

`deleteBills(billIds)` 逐条调用 `delete_bill`，但 state 中没有删除进度状态。批量删除 100 条时，用户无法知道当前进度（已删几条/共几条），只能等待全部完成。

建议增加 `deleting_progress: { current, total }` 状态。

- 严重程度：🟢 低
- 影响范围：用户体验

---

**问题 36：collection store 的 `active_tasks` 无上限和清理机制**

`active_tasks` 是 `{ task_id: { step, message, percent } }` 字典，任务完成后通过 `removeTask` 移除。但如果用户频繁触发解析任务（如连续点击解析），`active_tasks` 会累积大量条目。且 `removeTask` 仅在收到 `task_done` 事件时调用，若事件丢失则条目永不清理。

建议增加上限（如最多 10 个）和超时自动清理。

- 严重程度：🟢 低
- 影响范围：内存泄漏风险

### 9.3 组件设计

**问题 37：BillDetailDrawer 的 `visible` prop 双向绑定模式不明确**

组件规范中 `BillDetailDrawer` 的 Props 为 `billId + visible`，Events 为 `@close + @update`。但 Element Plus 的 `el-drawer` 使用 `v-model:modelValue` 控制显隐，文档未说明 `visible` 是否支持 `v-model` 或需要 `update:visible` 事件。建议明确为 `v-model:visible` 模式。

组件规范中这三个组件标注为"独立获取数据，内部管理状态"，但它们都依赖 accounting store。如果用户在三个 Tab 间切换，每次切换都会重新请求数据。建议在 accounting store 中缓存数据，Tab 切换时仅做增量刷新。

- 严重程度：🟢 低

**问题 39：CategoryPieChart 和 MonthlyBarChart 的交互事件参数不完整**

- `CategoryPieChart` emit `@slice-click(categoryId)`，但同一分类可能对应多个子分类，仅传 `categoryId` 无法区分
- `MonthlyBarChart` emit `@bar-click(date)`，但 `date` 的格式未定义（是 `2026-06` 还是 `2026-06-01`？），与 `billStore.filter.start_time/end_time` 的格式可能不匹配

- 影响范围：功能正确性

**问题 40：FileUploader 组件缺少文件类型校验和大小限制**

文档中拖拽上传区域标注"支持 .xlsx .csv .xls .zip"，但 `FileUploader` 组件的 Props/Events 规范中没有文件类型校验和大小限制参数。用户上传 `.pdf` 或超大文件时，错误只能由后端返回，前端无法提前拦截。

建议增加 `accept` prop 和 `max-size` prop。

### 9.4 Bridge 通信

**问题 41：Bridge `call()` 缺少超时机制**

`bridge.js` 的 `call()` 函数直接 `await api[method](params)`，没有超时控制。如果 Python 后端卡死，前端将永远等待。建议增加 `Promise.race` 超时机制（如 30 秒）。

- 严重程度：🟡 中
- 影响范围：用户体验

---

**问题 42：Bridge `on/off` 事件监听使用 `window.addEventListener` 但事件名可能冲突**

`on(eventName, handler)` 直接使用 `window.addEventListener`，事件名如 `task_progress`、`task_done` 是通用名称，可能与其他库或浏览器扩展的事件冲突。建议使用命名空间前缀（如 `pywebview:task_progress`）。

- 严重程度：🟢 低
- 影响范围：稳定性

---

**问题 43：Bridge `onTaskProgress` 返回的取消函数未在组件中保存**

文档示例中 `onTaskProgress` 返回取消函数，但使用示例没有保存它：

```javascript
onTaskProgress(task_id, onProgress, onDone)  // 返回值被丢弃
```

组件卸载时若不调用取消函数，会导致内存泄漏和重复回调。建议在 `onBeforeUnmount` 中调用取消函数。

- 严重程度：🟡 中
- 影响范围：内存泄漏

### 9.5 交互规范

**问题 44：删除确认弹窗规范与实际实现不一致**

交互规范中定义了三级确认弹窗（普通确认/危险操作/不可逆操作），但文档中所有删除操作（包括 `clear_all_bills`）都只使用了"危险操作"级别，没有"不可逆操作"级别（输入确认文字）的实例。`clear_all_bills` 作为最危险的操作，建议使用"不可逆操作"级别，要求用户输入"确认清除"等文字。

- 严重程度：🟢 低

**问题 45：批量操作缺少全选跨页支持**

BillListView 的批量删除使用 `el-table` 的多选功能，但 `el-table` 的全选只选择当前页。用户无法跨页全选。文档未定义跨页批量操作的需求和实现方式。

- 严重程度：🟢 低
- 影响范围：用户体验

### 10.1 账单采集→解析→入库流程

**问题 46：~~流程步骤 6（归属自动分配）与步骤 8（跨平台合并）存在执行顺序矛盾~~ ✅ 已修复**

**问题 47：流程步骤 7（信用消费识别）应在步骤 4（标准化）之后而非步骤 6 之后**

信用消费识别（花呗/白条标记）依赖 `trade_type` 和 `payment_method` 字段，这些字段在步骤 4 标准化时已确定。将信用识别放在步骤 6（归属分配）之后意味着信用标记的写入与归属分配在同一事务中，增加了事务复杂度。且信用识别不需要 `account_id`，逻辑上可以在归属分配之前执行。

虽然不影响正确性，但步骤顺序不够清晰，增加了理解成本。

- 严重程度：🟢 低
- 影响范围：设计清晰度

---

**问题 48：流程步骤 10（事务写入）的范围过大 — 全批导入在一个事务中**

文档说"整批导入使用数据库事务"，意味着一个采集记录中的所有账单（可能数百条）在一个事务中写入。如果其中一条在步骤 8 或 9 失败（如跨平台合并异常），整个批次回滚，包括已经成功处理的记录。

建议：按 N 条（如 50 条）分批提交事务，或对合并/配对异常做容错处理（跳过异常记录而非回滚整批）。

- 严重程度：🟡 中
- 影响范围：容错性

**问题 49：去重检查（步骤 5）仅依赖 (channel, channel_trade_no)，但建行记录无原生 trade_no**

建行账单的 `channel_trade_no` 是程序生成的 `CCB_{account_tag}_{YYYYMMDD}_{序号}`（问题 16），而非原始交易流水号。这意味着：

- 同一笔建行交易如果被导入两次（如用户重复上传文件），第二次会因序号不同而绕过去重检查
- 去重依赖的是生成策略的稳定性，而非数据的天然唯一性

建议：建行解析器应尽可能提取原始交易流水号，或增加基于（金额+时间+对方）的模糊去重机制。

- 严重程度：🟡 中
- 影响范围：去重可靠性

### 10.2 跨平台支付合并流程

**问题 50：场景 A 中合并后第三方记录 `is_deleted=1` 但 `bill_accounting` 仍存在 — 数据孤岛**

文档未定义 `is_deleted` 字段在 `unified_bills` 表中的 DDL（表结构中未列出此字段），也未定义其默认值和索引。

- 影响范围：数据完整性、查询正确性

**问题 51：场景 C 中银行卡记录扫描 orphan 的匹配条件不对称**

合并后保留银行卡记录，但其 `channel` 仍为 `ccb`。用户在账单列表中看到该记录时，渠道显示为"建行"，但实际交易发生在支付宝/微信。文档未说明合并后是否应更新 `channel` 字段或增加辅助显示字段。

### 10.3 转账配对流程

**问题 53：强匹配条件"备注含对方单号一致"过于严格**

强匹配要求"备注含对方单号一致"，但实际场景中：
- 微信转账备注通常不含支付宝单号
- 银行卡转账附言可能只有"转账"二字
- 跨渠道转账几乎不可能有对方单号

这导致强匹配几乎只在同渠道内生效（如微信→微信），跨渠道转账全部走弱匹配，增加用户确认负担。

建议：对跨渠道转账，增加基于"金额+时间+对方名称相似度"的中等置信度匹配，自动配对但可撤销。

- 严重程度：🟡 中
- 影响范围：用户体验

---

**问题 54：转账配对缺少金额容差**

文档中弱匹配条件为"金额相同"，但实际场景中：
- 微信转账 100 元，银行卡到账可能扣手续费后为 99.9 元
- 跨币种转账金额可能因汇率差异不完全一致

建议增加金额容差配置（如 ±0.1 元或 ±1%）。

- 严重程度：🟡 中
- 影响范围：功能正确性

**问题 55：`confirm_transfer_pair` 的 `transfer_link_id` 生成策略未定义**

API 返回 `{ success, transfer_link_id }`，但 `transfer_link_id` 的格式、生成规则、存储位置均未定义。`bill_accounting` 表中有 `transfer_link_id` 字段，但未说明是 UUID、自增 ID 还是其他格式。配对的两条记录是否共享同一个 `transfer_link_id`？如果是，如何保证唯一性？

- 严重程度：🟡 中
- 影响范围：数据模型完整性

### 10.4 重新解析流程

**问题 56：重新解析跳过 `is_manual_edited=1` 的记录可能导致分类不一致**

重新解析时跳过用户手动修改过的记录，但该记录的相邻记录（同一交易类型）可能被重新分类。例如：

- 原始：5 条"美团外卖"→ 分类"餐饮美食"（自动分类）
- 用户手动将其中 1 条改为"交通出行"
- 修改分类关键词后重新解析：4 条被重新分类为"外卖配送"（新分类），1 条仍为"交通出行"

结果：同一类型的交易出现不一致分类。文档未定义此场景的处理策略。

建议：重新解析后，对 `is_manual_edited=1` 的记录给出提示"以下记录的分类可能与新规则不一致，是否重新应用"。

- 严重程度：🟡 中
- 影响范围：数据一致性

---

**问题 57：重新解析仅更新 `unified_bills` 但不更新 `source_bills`**

流程中"从 source_bills 加载原始记录→重新解析+标准化→更新 unified_bills"，但 `source_bills` 表中的 `raw_json` 未更新。如果用户第二次重新解析，`source_bills` 中的原始数据仍是第一次解析的结果，而非真正的原始文件数据。

**问题 84：步骤 4（标准化）中"自动分类"依赖 `category_keywords`，但分类 ID 的有效性未校验**

标准化引擎 `Standardizer.auto_classify()` 根据 `category_keywords` 匹配后返回 `category_id`，但：

- 写入 `unified_bills.category_id` 时外键约束会失败，导致整批导入回滚

建议：`auto_classify` 返回前校验 `category_id` 是否存在，不存在则返回 `None`（未分类）。

- 影响范围：导入可靠性

**问题 85：步骤 6（账户自动创建）中微信/支付宝一个文件可能产生多个账户，但 `get_or_create_account` 的并发安全性未定义**

### 18.2 跨平台合并流程（7.2）

**问题 90：场景 B 中银行卡记录"正常入库（无需创建 bill_accounting 记录）"与场景 C 中"扫描 orphan 记录"的衔接存在逻辑缺口**

场景 B：银行卡记录导入 → 正常入库 → 无 `bill_accounting` 记录
场景 C：银行卡记录导入 → 扫描 orphan → 匹配成功 → 合并

但场景 C 是"银行卡导入时"执行扫描，这意味着场景 B 和场景 C 不可能同时发生（要么先有银行卡后有第三方，要么先有第三方后有银行卡）。问题在于：

- 场景 B 的银行卡记录没有 `bill_accounting`，后续如何被场景 A 的第三方记录匹配？
- 场景 A 的 `mark_orphan()` 搜索"金额一致+时间接近的银行卡记录"是全表扫描 `unified_bills`，不依赖 `bill_accounting`
- 但场景 C 的银行卡记录扫描 orphan 时，搜索的是 `bill_accounting.merge_status='orphan'`

两条搜索路径不对称：场景 A 搜索 `unified_bills`，场景 C 搜索 `bill_accounting`。如果场景 B 的银行卡记录后来被场景 A 的第三方记录匹配，匹配成功后需要为银行卡记录创建 `bill_accounting`（`merge_status='normal'`），但场景 A 的 `try_merge()` 流程中确实包含了"银行卡记录：在 bill_accounting 创建 merge_status='normal'"。

所以逻辑上是自洽的，但场景 B 的"无需创建 bill_accounting 记录"这个描述容易误导，建议改为"暂不创建 bill_accounting 记录，等待后续第三方记录匹配时按需创建"。

- 严重程度：🟢 低
- 影响范围：设计清晰度

**问题 91：场景 A 中"搜索银行卡记录"的搜索范围未限定渠道和账户**

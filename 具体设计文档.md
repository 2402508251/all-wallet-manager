# 本地统一账单工具 V1.0 具体设计文档

> 基于《本地统一账单工具 V1.0 功能需求文档》与当前代码实现整理。本文档用于说明系统架构、模块边界、核心数据模型和前后端交互方式，不展开过细字段、伪代码或未来设想。

---

## 一、文档说明

### 1.1 目标

本文档描述本地统一账单工具 V1.0 的当前设计，重点回答以下问题：

1. 系统如何运行在本地桌面环境中。
2. 前端、后端、数据库之间如何协作。
3. 账单从采集到报表的主流程如何组织。
4. 核心数据实体之间如何关联。
5. 当前代码中的主要页面和模块边界是什么。

### 1.2 设计原则

1. **本地优先**：账单、配置、快照均保存在本地，不依赖云端。
2. **轻量实现**：优先使用 Python 标准库、SQLite、Vue3 等轻量技术组合。
3. **源数据可追溯**：导入后的统一账单保留源账单 JSON，用于详情查看和重新解析。
4. **批量操作可回退**：重新解析、重新分类、批量归属调整等操作通过快照提供回退能力。
5. **逻辑删除优先**：普通删除进入回收站，永久删除才清理关联数据。
6. **事务保护**：批量写操作需要统一事务处理，避免部分成功导致数据不一致。

## 二、系统架构

### 2.1 总体架构

系统采用本地桌面单机架构：

| 层级 | 技术/模块 | 说明 |
| :--- | :--- | :--- |
| 桌面容器 | Python + PyWebView | 创建桌面窗口，加载前端页面，注册后端 API。 |
| 前端 | Vue3 + Vite + Pinia + Vue Router + Element Plus + ECharts | 提供页面、状态管理、表格表单和报表图表。 |
| 桥接层 | `window.pywebview.api` / `ApiBridge` | 前端通过 JS Bridge 调用 Python 后端能力。 |
| 后端应用层 | `api/bridge.py` | 对前端暴露统一 API，编排采集、解析、账务、报表、设置等服务。 |
| 领域模块 | `modules/*` | 采集、解析、标准化、归属、分类、账务、重新解析等业务模块。 |
| 数据层 | SQLite + `core/db.py` + `core/dal.py` | 本地数据库、初始化脚本、事务封装和通用数据访问。 |
| 快照层 | `core/snapshot.py` | 高影响操作前创建快照，支持查看和恢复。 |

启动入口为 `main.py`：

1. 初始化 `data/wallet.db`。
2. 创建 `DatabaseManager` 和 `ConfigManager`。
3. 创建并注册 `ApiBridge`。
4. 加载 `frontend/dist/index.html`；未构建时展示前端构建提示。
5. 创建 PyWebView 桌面窗口。

### 2.2 目录与模块概览

```text
AllWalletManager4/
├── main.py                    # PyWebView 桌面入口
├── api/bridge.py              # 前端调用的 API 门面
├── core/                      # 数据库、DAL、快照、配置、安全工具
├── modules/                   # 业务领域模块
├── config/                    # 枚举映射、邮箱白名单等配置
├── frontend/                  # Vue3 前端项目
└── data/wallet.db             # 运行时 SQLite 数据库
```

## 三、后端模块设计

### 3.1 API 门面层

`api/bridge.py` 是前端访问后端的统一入口。主要职责：

1. 提供统一返回结构：`{ success: bool, data: any, message: str }`。
2. 接收前端传入的 `params` 字典。
3. 编排 DAL、快照、配置和领域模块。
4. 对外提供采集、账单、账务、报表、设置、数据管理等 API 分组。
5. 维护部分轻量缓存和默认家庭/角色兜底逻辑。

### 3.2 数据与基础设施层

| 模块 | 主要职责 |
| :--- | :--- |
| `core/db.py` | SQLite 连接管理、建表、版本初始化、默认数据初始化。 |
| `core/dal.py` | 通用 CRUD、查询和事务封装。 |
| `core/snapshot.py` | 创建快照、记录变更明细、恢复快照、清理快照。 |
| `core/config_manager.py` | JSON 配置加载与保存。 |
| `core/crypto_utils.py` | 邮箱授权码等敏感信息加密。 |

数据库连接采用单一共享连接、全局锁和 WAL 模式，以降低 PyWebView 多线程调用下的 SQLite 文件锁冲突。

### 3.3 采集模块

目录：`modules/collection/`

| 文件 | 职责 |
| :--- | :--- |
| `file_upload.py` | 处理本地上传文件，保存到临时目录并生成采集记录。 |
| `email_fetch.py` | 通过 IMAP 主动拉取账单邮件和附件，维护增量拉取状态。 |
| `zip_handler.py` | 识别和解压 ZIP，支持加密 ZIP 密码流程。 |
| `channel_detector.py` | 根据文件名等信息自动识别微信、支付宝、建行渠道。 |

采集结果统一进入 `collection_records`，后续由解析流程读取。

### 3.4 解析与标准化模块

目录：`modules/parser/`、`modules/standardizer/`

| 模块 | 职责 |
| :--- | :--- |
| `parser_factory.py` | 根据渠道选择对应解析器。 |
| `wechat_parser.py` | 解析微信 `.xlsx` / `.csv` 账单。 |
| `alipay_parser.py` | 解析支付宝 `.csv` / `.xlsx` 账单。 |
| `ccb_parser.py` | 解析建设银行 `.xls` / `.pdf` 账单，并生成可去重交易标识。 |
| `field_mapper.py` | 将渠道原始字段映射为统一字段。 |
| `time_normalizer.py` | 将交易时间统一为系统时间格式。 |
| `amount_normalizer.py` | 将金额统一为整数分，并处理收支方向。 |
| `enum_mapper.py` | 将渠道枚举映射为系统内部枚举。 |

解析后形成统一账单，并保存源账单 JSON，便于详情展示和重新解析。

### 3.5 归属与账户模块

目录：`modules/attribution/`

| 文件 | 职责 |
| :--- | :--- |
| `account_extractor.py` | 从微信、支付宝、建行账单中提取账户标识和支付方式信息。 |
| `auto_assign.py` | 根据账户、角色、家庭关系进行自动归属。 |
| `manual_assign.py` | 支持手动归属调整相关查询和更新。 |

账户合并和微信别名维护由 `ApiBridge` 结合 `accounts`、`account_aliases` 等表实现。

### 3.6 分类模块

目录：`modules/categorizer/`

| 文件 | 职责 |
| :--- | :--- |
| `models.py` | 分类规则、匹配字段等领域对象。 |
| `rule_repository.py` | 分类、关键词、匹配字段的读取和保存。 |
| `field_resolver.py` | 构造分类匹配上下文，支持自身记录和发起方上下文。 |
| `scorer.py` | 根据关键词、字段、权重和优先级计算分类命中结果。 |
| `service.py` | 对单条账单执行分类。 |
| `reclassifier.py` | 对历史账单批量重新分类。 |

分类结果写入统一账单的分类字段，并记录分类来源、分数和规则信息。

### 3.7 账务处理模块

目录：`modules/accounting/`

| 文件 | 职责 |
| :--- | :--- |
| `cross_platform_merger.py` | 识别并处理跨平台支付、亲属卡/代付等真实支付者追踪。 |
| `transfer_pairer.py` | 识别转出/转入记录，生成转账关联，处理弱匹配确认。 |
| `credit_tracker.py` | 识别信用消费、还款流水和信用账户关联。 |
| `ccb_handler.py` | 建行账单特殊字段和内部交易标识辅助处理。 |

账务处理不删除原始账单，而是在 `bill_accounting` 中记录转账、信用、合并等状态。

### 3.8 重新解析与数据维护

目录：`modules/reparser/`

`reparser.py` 基于 `source_bills.raw_json` 使用当前解析和映射规则重新生成统一账单字段。重新解析前由快照模块创建备份，手动编辑过的账单默认避免被覆盖。

数据维护能力通过 `ApiBridge` 暴露，包括重新解析、重新分类、快照恢复、源账单清理、快照清理、按采集记录删除账单、清空账单等。

## 四、核心数据模型

### 4.1 账单主模型

| 实体 | 说明 |
| :--- | :--- |
| `unified_bills` | 统一账单主表，保存渠道、时间、类型、方向、金额、对方、商品说明、账户、角色、分类、删除状态、批次等核心信息。 |
| `source_bills` | 源账单表，保存原始记录 JSON，与统一账单一一关联。 |
| `bill_accounting` | 账务处理状态表，保存转账关联、信用状态、合并状态、真实支付者等信息。 |
| `import_batches` | 导入批次表，记录每次解析导入的统计结果。 |
| `collection_records` | 采集记录表，记录上传或邮箱拉取到的文件及解析状态。 |

### 4.2 归属模型

| 实体 | 说明 |
| :--- | :--- |
| `families` | 家庭维度，用于报表筛选和归属组织。 |
| `roles` | 角色维度，例如本人、配偶、公用角色等。 |
| `role_families` | 角色与家庭的关联关系。 |
| `accounts` | 账户表，保存渠道账户、银行卡、余额、信用相关账户等。 |
| `account_aliases` | 账户别名表，用于同一账户在不同账单中的名称归并。 |
| `credit_accounts` | 信用账户表，用于花呗、分付、信用卡等信用消费和还款追踪。 |

账单通常通过 `account_id` 关联到账户，再由账户关联到角色；家庭维度通过角色与家庭关系参与统计和归属管理。

### 4.3 分类模型

| 实体 | 说明 |
| :--- | :--- |
| `bill_categories` | 业务分类，支持层级结构和排序。 |
| `category_match_fields` | 可参与分类匹配的字段定义。 |
| `category_keywords` | 分类关键词规则，包含匹配字段、优先级、权重等信息。 |

分类模块根据规则为账单写入 `category_id`、分类来源和匹配分数。

### 4.4 快照模型

| 实体 | 说明 |
| :--- | :--- |
| `snapshots` | 快照主表，记录快照类型、描述、影响数量、创建时间等。 |
| `snapshot_details` | 快照明细，记录表、字段、旧值、新值等变更信息。 |

快照用于重新解析、重新分类、批量归属等操作后的回退，不替代完整数据库备份。

## 五、前后端交互设计

### 5.1 调用方式

前端通过 `frontend/src/utils/bridge.js` 封装 PyWebView 调用，最终访问 `window.pywebview.api` 暴露的方法。后端统一由 `ApiBridge` 接收参数并返回结果。

统一返回结构：

```js
{
  success: true | false,
  data: any,
  message: string
}
```

### 5.2 API 分组

| 分组 | 典型能力 |
| :--- | :--- |
| 采集与邮箱 | 文件选择、上传、ZIP 密码、解析采集记录、邮箱配置、测试连接、同步邮箱账单。 |
| 账单管理 | 查询账单、查看详情、更新账单、批量更新、删除、恢复、永久删除、回收站。 |
| 账务处理 | 孤立记录、合并记录、撤销合并、尝试合并、转账候选、确认/拒绝配对、信用账户。 |
| 报表统计 | 月度汇总、分类分布、趋势数据。 |
| 基础资料 | 家庭、角色、角色家庭关联、账户、账户别名、微信账户合并。 |
| 分类规则 | 分类维护、匹配字段、关键词保存、单条/批量重新分类。 |
| 数据管理 | 重新解析、快照列表、快照详情、快照恢复、源账单清理、快照清理、按采集记录清理。 |

### 5.3 耗时任务

上传、邮箱拉取、解析、重新解析等操作可能耗时较长。前端需要通过任务状态、进度面板或操作反馈提示用户当前处理阶段；后端写库部分需要通过事务保证一致性。

## 七、核心业务流程

### 7.1 导入解析流程

1. 用户上传文件或主动同步邮箱。
2. 系统生成采集记录并识别渠道。
3. 用户触发解析。
4. 系统处理 ZIP、选择解析器、解析原始记录。
5. 标准化时间、金额、交易类型、收支方向等字段。
6. 提取账户并执行自动归属。
7. 执行去重、分类和账务初步处理。
8. 写入统一账单、源账单、导入批次和采集记录结果。

### 7.2 账务处理流程

1. 信用追踪识别信用消费和还款流水。
2. 跨平台合并识别第三方支付与真实扣款记录。
3. 转账配对识别账户间转入转出。
4. 处理结果写入 `bill_accounting`，原始账单不被删除。
5. 报表根据账务状态控制统计口径。

### 7.3 数据维护流程

1. 用户发起重新解析、重新分类或批量调整。
2. 系统创建快照。
3. 系统执行批量变更。
4. 用户可查看结果，必要时通过快照恢复。
5. 用户可清理源账单、快照或指定采集记录关联账单。

## 八、设计约束与后续演进

### 8.1 当前约束

1. 数据库以本地 SQLite 为准，不引入远程数据库。
2. 数据库结构变更时优先维护初始化脚本，不做复杂历史迁移。
3. 需求不明确时不提前设计复杂功能，例如多家庭比例拆分、云同步、自动后台邮件轮询等。
4. 所有新增数据库写操作都需要考虑事务边界，避免嵌套事务。
5. 文档描述以当前代码为准，避免把尚未实现的设想写成已支持能力。

### 8.2 后续演进方向

后续可在保持轻量架构的前提下继续迭代：

1. 增强更多银行或平台账单解析器。
2. 完善分类规则可视化配置。
3. 增加更细的统计口径说明。
4. 增强导入错误诊断和用户修复提示。
5. 补充面向文档和核心业务流程的自动化验证脚本。

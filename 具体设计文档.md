### 1.2 进程通信机制

PyWebView 提供 JS ↔ Python 双向通信：

| 方向 | 机制 | 说明 |
|:---|:---|:---|
| JS → Python | `window.pywebview.api.方法名(参数)` | 前端调用后端暴露的 API，返回 Promise |
| Python → JS | `window.emit('event_name', data)` | 后端主动推送事件（进度、通知等） |

**API 注册方式**：Python 侧创建 Api 类，将所有方法注册到 `webview.api`，PyWebView 自动桥接到 JS 的 `window.pywebview.api`。

**异步处理约定**：
- 所有耗时操作（邮箱拉取、大文件解析）在 Python 侧使用 `threading.Thread` 执行
- 前端调用返回 `{ status: 'started', task_id: 'xxx' }` 后立即释放
- 后端通过 `window.emit('task_progress', { task_id, step, message })` 推送进度
- 后端通过 `window.emit('task_done', { task_id, result })` 通知完成

### 2.2 表结构定义

#### 2.2.1 families（家庭表）

| 字段 | 类型 | 约束 | 说明 |
|:---|:---|:---|:---|
| id | INTEGER | PK, AUTOINCREMENT | 主键 |
| name | TEXT | NOT NULL, UNIQUE | 家庭名称 |
| is_default | INTEGER | NOT NULL DEFAULT 0 | 是否默认家庭（0/1） |
| created_at | TEXT | NOT NULL DEFAULT CURRENT_TIMESTAMP | 创建时间 |

#### 2.2.2 roles（角色表）

| 字段 | 类型 | 约束 | 说明 |
|:---|:---|:---|:---|
| id | INTEGER | PK, AUTOINCREMENT | 主键 |
| name | TEXT | NOT NULL | 角色名称（如"本人"、"配偶"、"公用"） |
| role_type | TEXT | NOT NULL DEFAULT 'personal' | 角色类型：personal / shared |
| is_shared | INTEGER | NOT NULL DEFAULT 0 | 是否共享角色（联名卡等场景） |
| created_at | TEXT | NOT NULL DEFAULT CURRENT_TIMESTAMP | 创建时间 |

UNIQUE 约束：(name)

> **设计变更**：移除 `family_id` 字段。角色与家庭的关系改为多对多（FR-04-01 第3条："一个角色可以关联多个家庭"），
> 通过 `role_families` 关联表实现。共享角色可同时关联多个家庭，个人角色通常只关联一个家庭。

#### 2.2.2a role_families（角色-家庭关联表）

| 字段 | 类型 | 约束 | 说明 |
|:---|:---|:---|:---|
| role_id | INTEGER | NOT NULL, FK → roles.id | 角色ID |
| family_id | INTEGER | NOT NULL, FK → families.id | 家庭ID |
| is_primary | INTEGER | NOT NULL DEFAULT 0 | 是否主要家庭（每个角色必须有且仅一个 primary=1 的家庭） |

PRIMARY KEY：(role_id, family_id)

> **说明**：每个角色至少关联一个家庭（`is_primary=1`），共享角色可关联多个家庭。
> 创建角色时必须指定一个主要家庭，后续可添加更多家庭关联。

#### 2.2.3 accounts（资产账户表）

| 字段 | 类型 | 约束 | 说明 |
|:---|:---|:---|:---|
| id | INTEGER | PK, AUTOINCREMENT | 主键 |
| account_name | TEXT | NOT NULL | 账户名（如"微信-张三"、"建行尾号0480"） |
| account_tag | TEXT | | 账户标识（微信昵称/银行卡尾号） |
| channel | TEXT | NOT NULL | 渠道：wechat / alipay / ccb |
| role_id | INTEGER | NOT NULL, FK → roles.id | 所属角色 |
| balance_cents | INTEGER | NOT NULL DEFAULT 0 | 当前余额（分） |
| created_at | TEXT | NOT NULL DEFAULT CURRENT_TIMESTAMP | 创建时间 |

UNIQUE 约束：(account_tag, channel)

#### 2.2.4 credit_accounts（信用账户表）

| 字段 | 类型 | 约束 | 说明 |
|:---|:---|:---|:---|

| account_name | TEXT | NOT NULL | 信用账户名（如"花呗"、"京东白条"） |

#### 2.2.5 unified_bills（统一账单表 — 核心表）

> 已去除 `balance_cents`、`counterparty_account`、`channel_merchant_no` 三个渠道独有字段，
> 其完整数据已存在于 `source_bills.raw_json` 中，查看详情时可从源账单取回。
> 账务处理状态字段已拆分至 `bill_accounting` 表（见 2.2.6）。
> **归属分配设计**：角色与家庭为多对多关系（FR-04-01），账单需同时记录 `role_id` 和 `family_id`。
> 账单不可手动分配角色/家庭。`role_id` 从 `account_id → accounts.role_id` 自动派生，
> `family_id` 从角色的主要家庭（`role_families.is_primary=1`）自动填充。
> FR-04-02 的"手动分配"通过系统设置页的账户管理实现：用户将账户分配给角色，账单导入时自动匹配账户获得角色和家庭。
> `assign_status='pending'` 的账单需用户在设置中为对应账户指定角色后触发重新分配。
>
> **共享角色跨家庭账单分配策略**：
> - 共享角色（role_type='shared'）可关联多个家庭，但每条账单只能归属一个家庭
> - 默认策略：账单的 family_id 取角色的主要家庭（is_primary=1），这是最简单的方案
> - 用户可手动调整：在账单详情中提供"变更归属家庭"按钮，允许将共享角色下的账单
>   重新分配到角色的其他关联家庭（仅限该角色已关联的家庭列表中选择）
> - 不支持按比例分摊：一条账单只能归属一个家庭，不支持"50%归A家庭、50%归B家庭"
>   这种分摊逻辑过于复杂，超出当前需求范围

### 3.3 金额标准化规则

```
输入：原始金额字符串（如 "4.5"、"-2,073.00"、"¥20.00"）
输出：amount_cents（整数分）+ direction（方向）

规则：
1. 去除所有非数字字符（¥、逗号、空格等），保留负号和小数点
2. 使用 Decimal 精确计算：amount_cents = int(Decimal(cleaned_value) * 100)
3. 若原始金额为负数：direction = 'expense'，amount_cents 取绝对值
4. 若原始金额为正数：direction = 'income'，amount_cents 保持正值
5. 若原始金额为零：direction = 'neutral'
6. 微信/支付宝：金额本身不含正负号，方向由"收/支"字段决定
7. 建行：金额含正负号，方向由正负判定
```

### 3.4 时间标准化规则

| 渠道 | 原始格式 | 标准化输出 |
|:---|:---|:---|
| 微信 | datetime 对象（openpyxl 读取） | `2026-06-01T20:03:56+08:00` |
| 支付宝 | `"2026/6/6 12:43"` 字符串 | `2026-06-06T12:43:00+08:00` |
| 建行 | `"20250625"` 字符串 | `2025-06-25T00:00:00+08:00` |

统一输出格式：ISO 8601 带时区 `YYYY-MM-DDTHH:MM:SS+08:00`

#### 4.1.3 ZIP 处理（zip_handler.py）

```python
class ZipHandler:
    def detect_encryption(self, file_path: str) -> dict
        """
        检测 ZIP 文件加密类型：
        1. 使用 zipfile 读取 ZIP 文件头信息
        2. 检查压缩方法的标志位：
           - 0x01（加密标志）+ 0x63（AES 加密标志）→ encryption_type='aes'
           - 0x01（加密标志）无 AES 标志 → encryption_type='zipcrypto'
           - 无加密标志 → encryption_type='none'
        3. 返回 { encrypted: bool, encryption_type: 'aes'|'zipcrypto'|'none' }

        注意：微信/支付宝导出的 ZIP 使用 AES-256 加密，
        Python 标准库 zipfile 不支持 AES，需使用 pyzipper 库
        """

    def extract(self, file_path: str, password: str = None) -> list[str]
        """
        解压 ZIP 文件：
        1. 先调用 detect_encryption 检测加密类型
        2. 根据加密类型选择解压策略：
           - encryption_type='none' → zipfile 标准解压（无需密码）
           - encryption_type='zipcrypto' → zipfile 解压（需密码）
           - encryption_type='aes' → pyzipper 解压（需密码，AES-256）
        3. 若加密但未提供密码 → 返回 { need_password: True }
        4. 若密码错误 → 返回 { error: 'password_incorrect' }
        5. 若加密类型不支持 → 返回 { error: 'unsupported_encryption' }
        6. 解压成功 → 返回解压后的文件路径列表
        7. 解压后自动清理原始 ZIP（仅保留解压产物）

        依赖：pyzipper（pip install pyzipper，需在 requirements.txt 中添加）
        支持的加密算法：AES-256（微信/支付宝）、ZipCrypto（传统 ZIP 加密）
        """

    def cleanup_temp(self, file_path: str) -> None
        """清理临时文件"""
```

from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class ParseResult:

    success: int

#### 4.2.4 建行解析器（ccb_parser.py）

```python
class CCBParser(BaseParser):
    def parse(self, file_path: str) -> ParseResult:
        """
        解析建设银行账单（自动检测格式）：
        1. 根据文件扩展名选择解析策略：
           - .xls → _parse_xls()
           - .pdf → _parse_pdf()
        2. 返回 ParseResult
        """

    def _parse_xls(self, file_path: str) -> ParseResult:
        """
        解析建设银行 xls 账单：
        1. xlrd 打开工作簿
        2. 跳过前 3 行元信息，第 3 行为表头
        3. 从第 4 行开始读取数据
        4. 自动生成 channel_trade_no（CCB_{account_tag}_{date}_{seq}）
        5. 金额标准化（含正负号、千分位逗号）
        6. 对方账号与户名按 "/" 分割
        7. 区分交易日期与记账日期（统一用交易日期）
        8. 返回 ParseResult
        """

    def _parse_pdf(self, file_path: str) -> ParseResult:
        """
        解析建设银行 PDF 账单：
        1. 使用 pdfplumber 打开 PDF 文件
        2. 定位交易明细表格区域：
           - 查找包含"交易日期"的行作为表头
           - 提取表头列名（交易日期/记账日期/交易描述/金额/余额等）
        3. 逐行提取表格数据：
           - 处理跨行合并单元格（pdfplumber 的 extract_tables 返回嵌套列表）
           - 跳过空行和汇总行（如"合计"/"小计"）
        4. 字段映射与标准化：
           - 交易日期 → trade_time（格式：YYYY-MM-DD，PDF 中可能为 YYYY/MM/DD）
           - 交易描述 → product_desc + counterparty（按空格或特定分隔符拆分）
           - 金额 → amount_cents（注意 PDF 中金额可能含千分位逗号和正负号）
           - 余额 → 忽略（不存储）
        5. 自动生成 channel_trade_no（CCB_PDF_{account_tag}_{date}_{seq}）
        6. direction 判定：金额为正 → income，金额为负 → expense
        7. 返回 ParseResult

        PDF 解析注意事项：
        - pdfplumber 依赖：pip install pdfplumber（需在 requirements.txt 中添加）
        - 编码问题：建行 PDF 可能使用 GBK 编码，pdfplumber 默认 UTF-8，
          需设置 laparams 或使用 fallback 编码
        - 表格边界：部分 PDF 账单的表格可能跨页，需合并多页的表格数据
        - 容错：若 pdfplumber 无法提取表格（扫描件 PDF），返回空结果并标记
          collection_record.status='parse_failed'，提示用户手动上传 xls
        """
```

    def extract_ccb_account_info(self, file_path: str) -> dict:

        建行账单无账户标识信息，返回空
        需用户通过 set_account_tag 手动指定
        """

    def extract(self, channel: str, file_path: str) -> dict:

**账户标识格式规范**：

| 渠道 | 格式 | 示例 |
|:---|:---|:---|
| 微信 | `wechat-{昵称}-{支付方式}` | `wechat-风-零钱`、`wechat-风-招商银行(1234)` |
| 支付宝 | `alipay-{银行卡尾号}-{银行名}` | `alipay-0480-建设银行储蓄卡`、`alipay-余额-账户余额` |
| 建行 | `ccb-{用户指定}` | `ccb-0480`（需手动指定） |

**解析流程变更**：

1. 解析账单前，调用 `AccountExtractor.extract()` 提取账户信息
2. 为每种支付方式调用 `get_or_create_account()` 创建账户（`role_id=NULL`，表示尚未分配角色）
3. 每条账单根据 `payment_method` 匹配对应账户，设置 `account_id`
4. 账单 `role_id=NULL`，`family_id=NULL`，`assign_status='pending'`（等待用户为账户指定角色）

**默认家庭和角色**（FR-04-01 第1条）：

系统自动创建"未分配"家庭和角色，作为用户分配角色时的兜底选项：
- 家庭名：`未分配`，`is_default=1`
- 角色名：`未分配`，`role_type='personal'`
- 角色与未分配家庭关联：`role_families.is_primary=1`

> **assign_status 语义澄清**：
> - `assigned`：账户已关联真实角色（非"未分配"角色），账单的 `role_id` 和 `family_id` 从账户和角色自动派生
> - `pending`：账户尚未关联角色（`role_id=NULL`），或账户关联的是"未分配"角色，账单的 `role_id` 和 `family_id` 为 NULL
>
> 用户在设置页为账户指定角色后，调用 `reassign_pending_bills` 将 `assign_status` 从 `pending` 更新为 `assigned`，
> 同时回填 `role_id` 和 `family_id`。若用户将账户角色改回"未分配"角色或清空角色，
> 则该账户下所有账单的 `assign_status` 回退为 `pending`，`role_id` 和 `family_id` 清空。
>
> 前端展示逻辑：`assign_status='pending'` 的账单在列表中显示"待分配"标签，
> 提示用户前往设置页为对应账户指定角色。

        """

    def auto_pair_weak_candidates(self, record: dict) -> list[dict]:
        """
        弱匹配候选搜索：
        条件：金额相同 + 方向相反 + 时间差满足规则 + 对方名称相似
          - 即时渠道（微信→零钱/支付宝→余额）：≤5分钟
          - 跨行渠道：≤1天
        对方名称相似度算法（三级匹配，满足任一即通过）：
          1. 精确包含：一方名称完全包含另一方（如"张三"包含于"张三（转账）"）
          2. 去括号后精确匹配：去除括号及括号内内容后字符串相等
             （如"张三（转账）"→"张三"，与"张三"匹配）
          3. 编辑距离阈值：Levenshtein 距离 ≤ max(2, min(len_a, len_b) * 0.3)
             （容忍少量字符差异，如"张三丰"与"张三"距离为1，通过）
        结果：返回候选列表，禁止自动配对（即使只有一条候选也需用户确认）
        """

    def confirm_pair(self, out_id: int, in_id: int, fee_amount: int = 0) -> dict:
        """
        用户确认配对：
        1. 生成 transfer_link_id（UUID）
        2. 在两条记录的 bill_accounting 中写入 transfer_link_id
        3. 若 fee_amount > 0，调用 generate_fee_record 生成手续费记录

        手续费触发条件（满足任一即触发）：
        a. 用户在配对确认弹窗中手动输入手续费金额（fee_amount > 0）
        b. 转出金额与转入金额的差额即为手续费（自动计算）：
           fee_amount = out_record.amount_cents - in_record.amount_cents
           仅当差额 > 0 且 ≤ 合理范围（≤ 转出金额的 5%）时自动填入
        c. 若账单中已存在独立的 trade_type='fee' 记录（如微信提现手续费），
           不再生成重复记录，而是将该已有记录的 bill_accounting.transfer_link_id
           关联到当前配对
        """

    def generate_fee_record(self, transfer_record: dict, fee_amount: int) -> dict:
        """
        生成手续费记录：
        - unified_bills: trade_type='fee', direction='expense', is_system=1
        - bill_accounting: 写入同一 transfer_link_id
        - channel_trade_no: 'FEE_' + 原始 channel_trade_no
        - amount_cents: fee_amount（用户输入或自动计算的差额）
        - source_bill_id: NULL（系统生成）
        - 统计查询默认排除 is_system=1 的记录

        金额来源优先级：
        1. 用户手动输入（最权威）
        2. 转出-转入差额自动计算（需确认差额合理）
        3. 不从原始账单中提取（原始账单中的手续费是独立记录，应关联而非重复生成）
        """

### 4.6 全局快照引擎（snapshot.py）

```python
class SnapshotEngine:
    def create_snapshot(self, snapshot_type: str, description: str,
                        affected_records: list[int], batch_id: str = None) -> int:
        """
        创建快照（操作前调用，全量备份模式，两表存储）：

        设计原则：快照采用"全量备份"语义而非"增量 Diff"语义，确保回退时可以精确恢复
        操作前的完整状态。即使某些字段未变更，也记录其操作前的值，防止遗漏。

        流程：
        1. 读取 affected_records 对应的当前数据
           - unified_bills 主表数据（所有字段，含 is_deleted/is_system 等标记字段）
           - bill_accounting 扩展表数据（若存在）
        2. 创建 snapshot 记录，写入 snapshots 表
        3. 逐表逐条记录**所有字段**的当前值，写入 snapshot_details 表
           - table_name 区分来源表：unified_bills / bill_accounting
           - old_value 记录当前值（操作前的值）— 全量记录，不跳过任何字段
           - new_value 暂时为 NULL，等待操作完成后由 finalize_snapshot 回填
        4. 返回 snapshot_id

        与增量 Diff 模式的区别：
        - Diff 模式：仅记录变更字段，回退时只恢复变更部分，若遗漏字段则回退不完整
        - 全量备份模式：记录所有字段，回退时恢复完整行数据，确保精确回退
        - 存储开销：全量备份占用更多空间，但快照通常保留数量有限（默认50条），可接受
        """

    def finalize_snapshot(self, snapshot_id: int, affected_records: list[int]) -> None:
        """
        完成快照（操作后调用，回填 new_value 并标记变更状态）：

        流程：
        1. 读取 affected_records 对应的当前数据（操作后的值）
        2. 逐条比对 snapshot_details 中的 old_value 与当前值
        3. 将当前值写入 new_value
        4. 标记变更状态：
           - old_value == new_value 的记录保留（不删除），用于回退时确认字段未被修改
           - 可通过 is_changed 字段（或查询时比对 old_value != new_value）区分变更/未变更字段
        5. 对于操作后被物理删除的记录（affected_records 中存在但数据库中不存在），
           在 snapshot_details 中标记 is_deleted_after=1，回退时需重新插入

        注意：此方法必须在操作完成、事务提交后调用
        """

    def restore_snapshot(self, snapshot_id: int) -> dict:
        """
        回退快照（全量恢复模式）：

        设计原则：回退时恢复快照记录的所有字段的 old_value，而非仅恢复变更字段。
        这确保了即使某些字段在操作后被意外修改（未被 Diff 捕获），也能恢复到操作前的精确状态。

        流程：
        1. 读取 snapshot_details（按 table_name 分组）
        2. 逐表逐条将 old_value 恢复到对应表
           - unified_bills 表字段 → UPDATE unified_bills SET field_name = old_value WHERE id = bill_id
           - bill_accounting 表字段 → UPDATE bill_accounting SET field_name = old_value WHERE bill_id = bill_id
        3. 对于操作后被物理删除的记录（is_deleted_after=1）：
           - 尝试重新插入完整行（从 snapshot_details 中的 old_value 重建）
           - 若外键冲突则跳过并记录警告
        4. 对于操作后新增的记录（不在 snapshot_details 中但存在于数据库中）：
           - 删除这些新增记录（它们是操作产生的，回退时应移除）
           - 仅删除 is_system=1 的系统生成记录（镜像/手续费），不删除用户数据
        5. 使用事务保证原子性
        6. 返回恢复结果统计
        """

    def get_snapshot_details(self, snapshot_id: int) -> list[dict]:
        """
        获取快照变更明细：
        返回 snapshot_details 列表，每条含：
        bill_id, table_name, field_name, old_value, new_value
        """

    def list_snapshots(self, limit: int = 20) -> list[dict]:
        """列出最近的快照"""

    def delete_snapshot(self, snapshot_id: int) -> bool:
        """
        删除快照（释放空间）：
        同时删除 snapshot 和关联的 snapshot_details
        """

    def cleanup_old_snapshots(self, keep_count: int = 50) -> int:
        """
        清理旧快照：
        保留最近 keep_count 条，删除更早的（含关联 snapshot_details）
        返回删除条数
        """
```

### 4.7 重新解析引擎（reparser.py）

```python
class ReparseChange:
    """记录单条账单重新解析前后的字段变更"""
    bill_id: int
    old_values: dict          # 变更前的字段值
    new_values: dict          # 变更后的字段值
    changed_fields: set[str]  # 实际变更的字段名集合

class ReParser:
    def reparse(self, scope: dict, config_manager: ConfigManager) -> dict:
        """
        重新解析：
        1. 根据 scope 确定范围（全部/指定批次/指定渠道/指定时间范围）
        2. 创建全局快照（调用 SnapshotEngine.create_snapshot，记录 old_value）
        3. 从 source_bills 加载原始记录
        4. 重新走解析+标准化流程
        5. 仅更新可变字段（trade_type, category_id, direction, counterparty, product_desc, amount_cents）
        6. 跳过 is_manual_edited=1 的记录（不覆盖用户手动修改的分类等字段）
        7. 逐条比对变更，收集 ReparseChange 列表
        8. 调用 _cascade_accounting_updates 处理账务处理状态级联更新
        9. 使用事务保证原子性
        10. 完成快照（调用 SnapshotEngine.finalize_snapshot，回填 new_value）
        11. 返回变更统计（含级联影响统计）

        scope 格式：
        {
            "type": "all" | "batch" | "channel" | "time_range",
            "batch_id": "xxx",        # type=batch 时
            "channel": "wechat",      # type=channel 时
            "start_time": "...",      # type=time_range 时
            "end_time": "..."         # type=time_range 时
        }
        """

    def _cascade_accounting_updates(self, changes: list[ReparseChange]) -> dict:
        """
        重新解析后对 bill_accounting 的级联更新：

        核心原则：重新解析仅更新 unified_bills 的可变字段，bill_accounting 的状态
        需要根据字段变更类型进行级联调整，确保数据一致性。

        ┌──────────────────────────────────────────────────────────────────────┐
        │                     级联更新规则矩阵                                  │
        ├──────────────────┬───────────────────────────────────────────────────┤
        │ 变更字段         │ 级联动作                                          │
        ├──────────────────┼───────────────────────────────────────────────────┤
        │ trade_type 变更  │ 1. 信用消费标记级联（见规则 A）                    │
        │                  │ 2. 合并/配对关系解除（见规则 B/C）                 │
        │ amount_cents 变更│ 1. 合并关系解除（见规则 B）                       │
        │                  │ 2. 转账配对关系解除（见规则 C）                    │
        │ counterparty 变更│ 1. 合并记录的 original_counterparty 不受影响       │
        │                  │   （保留合并时的原始值）                           │
        │ product_desc 变更│ 1. 合并记录的 original_product_desc 不受影响       │
        │                  │   （保留合并时的原始值）                           │
        │ category_id 变更 │ 无级联影响（bill_accounting 不涉及分类）           │
        │ direction 变更   │ 无级联影响（direction 由 trade_type 决定，         │
        │                  │   正常流程中不会单独变更）                         │
        └──────────────────┴───────────────────────────────────────────────────┘

        规则 A：信用消费标记级联
        ─────────────────────────
        当 trade_type 发生以下变更时：

        A1. consumption → credit_consumption：
            - 若 bill_accounting 不存在 → 创建记录，is_credit=1
            - 若 bill_accounting 存在且 is_credit=0 → 更新 is_credit=1
            - 调用 CreditTracker.identify_credit 重新识别，填充 credit_account_id

        A2. credit_consumption → consumption：
            - 若 bill_accounting.is_credit=1 → 更新 is_credit=0，清空 credit_account_id
            - 注意：不删除 bill_accounting 记录（可能还有其他状态字段）

        A3. 其他类型 → credit_consumption：
            - 同 A1

        A4. credit_consumption → 其他类型（非 consumption）：
            - 若新类型为 repayment → 保留 is_credit 相关信息，交由 CreditTracker.link_repayment 处理
            - 若新类型为 transfer_out/transfer_in → 清空 is_credit 和 credit_account_id

        规则 B：跨平台合并关系级联
        ─────────────────────────
        当 trade_type 或 amount_cents 发生变更时：

        B1. 已合并的银行卡记录（merge_status='normal'，且有 merged_to_id 指向它的记录）：
            - amount_cents 变更 → 解除合并关系：
              a. 恢复第三方记录：is_deleted=0，恢复 role_id/family_id（从 account_id 派生），
                 bill_accounting.merge_status 改为 'normal'
              b. 恢复银行卡记录的原始 counterparty/product_desc：
                 从 original_counterparty/original_product_desc 读取并回写
              c. 删除银行卡记录的 bill_accounting 记录（恢复为无账务状态）
            - trade_type 变更（如从 consumption 变为 transfer_out）→ 同上解除合并

        B2. 已合并的第三方记录（merge_status='merged_source'，is_deleted=1）：
            - 此记录已被逻辑删除，重新解析不会处理 is_deleted=1 的记录
            - 但如果通过 undo_merge 恢复后再重新解析，则按新 trade_type 处理

        B3. 孤儿记录（merge_status='orphan'）：
            - amount_cents 变更 → 更新 unified_bills.amount_cents，
              orphan 记录的匹配条件已变，下次合并扫描时可能匹配到不同的银行卡记录
            - trade_type 变更为非消费类型 → 解除孤儿状态：
              删除 bill_accounting 记录（非消费类型不参与跨平台合并）

        规则 C：转账配对关系级联
        ─────────────────────────
        当 trade_type 或 amount_cents 发生变更时：

        C1. trade_type 不再是 transfer_out/transfer_in：
            - 若存在 transfer_link_id → 解除配对：
              a. 清空本条记录的 transfer_link_id
              b. 查找配对记录（相同 transfer_link_id 的另一条），
                 清空其 transfer_link_id
              c. 若配对记录有手续费记录（相同 transfer_link_id + trade_type='fee'），
                 删除手续费记录（unified_bills + bill_accounting）

        C2. trade_type 仍为 transfer_out/transfer_in，但 amount_cents 变更：
            - 若存在 transfer_link_id → 解除配对（金额不再匹配）：
              a. 同 C1 的 a/b/c 步骤
              b. 解除后，两条记录将成为未配对状态，可重新参与自动/手动配对

        C3. trade_type 变为 transfer_out/transfer_in（原非转账类型）：
            - 无既有配对关系，但可参与后续转账配对检查
            - 不自动触发配对（需用户在账务处理页手动操作或下次导入时触发）

        返回值：
        {
            "credit_updates": int,      # 信用标记更新数
            "merge_cascades": int,      # 合并关系级联处理数
            "pair_cascades": int,       # 配对关系级联处理数
            "unpaired_bills": list[int] # 被解除配对的账单ID列表（供前端提示）
        }
        """
```

### 5.6 Bridge 通信封装（bridge.js）

```javascript
// utils/bridge.js

// PyWebView API 是否就绪检测
export function isReady() {
  return !!window.pywebview?.api
}

// 等待 PyWebView API 就绪（Vue app mount 前调用）
export function ready() {
  return new Promise((resolve) => {
    if (window.pywebview?.api) return resolve()
    window.addEventListener('pywebviewready', () => resolve())
  })
}

// 通用 API 调用封装（统一错误处理 + 返回值解包）
export async function call(method, params = {}) {
  const api = window.pywebview?.api
  if (!api || !api[method]) {
    throw new Error(`API 方法不可用: ${method}`)
  }
  const result = await api[method](params)
  if (result.success === false) {
    throw new Error(result.message || '操作失败')
  }
  return result.data
}

// 事件监听封装
export function on(eventName, handler) {
  window.addEventListener(eventName, (event) => handler(event.detail))
}

export function off(eventName, handler) {
  window.removeEventListener(eventName, (event) => handler(event.detail))
}

// 便捷方法：监听任务进度
export function onTaskProgress(taskId, onProgress, onDone) {
  const progressHandler = (data) => {
    if (data.task_id === taskId) onProgress(data)
  }
  const doneHandler = (data) => {
    if (data.task_id === taskId) {
      off('task_progress', progressHandler)
      off('task_done', doneHandler)
      onDone(data)
    }
  }
  on('task_progress', progressHandler)
  on('task_done', doneHandler)
  return () => {
    off('task_progress', progressHandler)
    off('task_done', doneHandler)
  }
}
```

// composables/useECharts.js
import { ref, onMounted, onBeforeUnmount, watch } from 'vue'
import { echarts } from '@/utils/echarts'

export function useECharts(chartRef, optionGetter) {
  let chart = null

  const initChart = () => {
    if (chartRef.value) {
      chart = echarts.init(chartRef.value)
      chart.setOption(optionGetter())
    }
  }

  const updateChart = () => {
    if (chart) chart.setOption(optionGetter())
  }

  const handleResize = () => chart?.resize()

  onMounted(() => {
    initChart()
    window.addEventListener('resize', handleResize)
  })

  onBeforeUnmount(() => {
    window.removeEventListener('resize', handleResize)
    chart?.dispose()
  })

  return { updateChart, handleResize }
}

| 方法名 | 参数 | 返回 | 说明 |
|:---|:---|:---|:---|
| `query_bills` | `{ filters: { channel, family_id, role_id, category_id, direction, assign_status, merge_status, start_time, end_time, keyword, is_deleted }, page, page_size }` | `{ total, list: [...] }` | 多条件查询账单（merge_status 需 LEFT JOIN bill_accounting）。is_deleted 默认为 0（仅查询未删除记录），传 1 查询回收站 |

> **删除策略说明**：
> - `delete_bill` 为逻辑删除（`is_deleted=1`），账单仍在数据库中，所有查询默认过滤 `is_deleted=0`，不显示已删除记录

> - `permanent_delete_bill` 为物理删除，会级联清理所有关联数据，操作不可逆，需前端二次确认
> - 批量删除通过逐条调用 `delete_bill` 实现（逻辑删除），不提供批量物理删除
>
> **归属分配说明**：账单的 role_id/family_id 不可直接修改，从 account_id 自动派生。FR-04-02 的"手动分配"通过两种方式实现：
> 1. **账户管理**（推荐）：在设置页将账户分配给角色，该账户下所有账单自动级联更新
> 2. **批量修改账户**（FR-04-02 第4条）：在账单列表中选中多条账单，通过 `batch_reassign_bills` 批量修改其 account_id，间接改变角色和家庭归属
